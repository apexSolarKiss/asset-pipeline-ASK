# Truth Pressure Across Workflow Modes Note

## Purpose

This note compares `single-item truth`, `relational truth`, and `campaign coherence` across the current worked-example tracks.

Its purpose is to state only what that three-way comparison now supports structurally without turning the comparison into schema branching, taxonomy settlement, or new seam logic.

## Why This Three-Way Comparison Is Now Possible

The repo now has three worked examples that apply distinct governing pressure inside the same basic packet-and-bridge structure.

That makes it possible to compare not just tighter and looser workflow modes, but three different ways truth and coherence can bear on the workflow without assuming they require different seam systems.

## Single-Item Truth

In the SKU-driven case, truth pressure concentrates on one item and its repeatable comparability.

The strongest governing burden is that one product remains legible, faithful, and consistent enough to support controlled downstream use across repeated outputs. This is the tightest current case for item-level truth and comparability.

## Relational Truth

In the collection / merchandising case, truth pressure concentrates on governed relationships across multiple items.

The strongest governing burden is not one hero SKU alone, but whether adjacency, spacing, pairing, and positional emphasis keep the grouped set legible as one coherent merchandising arrangement. This is a relationship-bearing truth pressure rather than a single-item one.

## Campaign Coherence

In the campaign / editorial case, pressure concentrates on bounded coherence across a broader expressive range.

The strongest governing burden is whether multiple assets remain recognizably part of one campaign family while preserving required signals and staying within the approved directional frame. This is the loosest current case, but it is still bounded rather than unconstrained.

## What Still Appears Shared Across The Three

Despite those differences, the current examples still appear to share one structural pattern.

- inputs and context establish what the work is about
- constraints bound what later work must remain within
- governed outputs must be sufficient for controlled downstream use
- approval-facing readiness remains distinct from automated approval itself

The strongest current claim is not that the three modes use the same truth pressure, but that they place different governing pressures inside one shared structure.

## What This Comparison Does Not Justify

This comparison does not by itself justify:

- new seams
- schema branching
- generalized workflow taxonomy
- a stronger claim that one example has now become the model for all others

## What This Note Does Not Do

- define field catalogs
- formalize schema
- introduce runtime or orchestration language
- imply validator or code changes
- settle workflow taxonomy beyond current repo-local truth
