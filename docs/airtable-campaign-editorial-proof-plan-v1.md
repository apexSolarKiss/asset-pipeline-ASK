# Airtable Campaign / Editorial Proof Plan v1

## Purpose Of This Proof

This document defines the smallest honest next Airtable proof after the SKU-driven Furniture v1 chain.

Its purpose is to test whether the current Airtable proof shape can carry one bounded campaign / editorial burden at all before any schema revision, Airtable configuration revision, or broader mode rollout is considered.

This is a Cycle-1-equivalent planning artifact for a second workflow mode. It is not a campaign/editorial system design and it is not an implementation move.

## Why Campaign / Editorial Is The Next Honest Mode

SKU-driven Furniture v1 has already been pressure-tested through multiple bounded cycles.

That work established:

- the current seven-table Airtable proof can carry one bounded SKU-mode packet and seam chain
- the proof can survive denser packet burden without graph inadequacy appearing first
- the prompt-artifact continuity question is now explicitly on hold pending later trigger conditions

That means the next honest question is no longer how to keep refining SKU mode.

The next honest question is whether the current proof shape can carry a materially different workflow mode at all.

Campaign / editorial is the strongest next candidate because current repo-local truth already suggests that it carries:

- heavier review-artifact burden
- more family-level judgment across multiple assets
- looser but still bounded coherence pressure than SKU-driven product imagery

This makes campaign / editorial the smallest honest next mode test without turning the repo toward generalized second-mode rollout.

## Bounded Scenario

The proof should use one narrow campaign / editorial scenario only:

- one approved campaign direction
- one small branded story world
- one three-asset mini-family that must read as one campaign family

The bounded scenario should be framed as one hero asset plus two companion assets, where the goal is not full campaign coverage but bounded family coherence under reviewable conditions.

The proof should stay small enough that failure can be interpreted as a proof-shape signal rather than as scale noise.

## Asset-Family Burden

The candidate burden for this proof is:

- one hero asset that establishes the family read
- two companion assets that must remain recognizably part of the same family
- bounded directional context that governs campaign tone, expressive range, and coherence

This is materially different from the SKU-driven Furniture v1 burden.

In SKU mode, the governing pressure sits first on one product remaining faithful, legible, and repeatably usable.

In this campaign / editorial proof, the governing pressure sits first on whether multiple assets remain legibly part of one bounded family while still preserving required directional signals.

The proof should therefore treat family coherence as the primary burden, not product truth alone.

## Role Of The Review Board

This proof should treat one side-by-side review board plus light annotation as explicit load-bearing review context.

That does not make the review board a generalized approval system and it does not create a new seam.

It means only that, for this bounded campaign / editorial proof, reviewable side-by-side context is part of the burden being tested because isolated outputs are less likely to show family coherence honestly.

The planned review artifact burden is:

- one side-by-side review board spanning the hero asset and both companion assets
- one light annotation or review note layer sufficient to explain the intended family read and directional consistency

This is the smallest honest campaign-side review burden already supported by repo-local mode notes.

## How This Differs From SKU-Driven Furniture v1

The difference from SKU-driven Furniture v1 is not that campaign / editorial needs an entirely different system.

The difference is that this proof asks the current Airtable shape to carry a different governing burden:

- SKU mode centers single-item truth and repeatable comparability
- campaign / editorial centers bounded coherence across a mini-family
- SKU mode can treat review artifacts as useful but secondary
- campaign / editorial appears to make review artifacts more structurally load-bearing for readiness-supporting context

This is therefore a mode-difference proof, not a claim that campaign / editorial has become the dominant workflow model.

## First Seam Trail To Test

The first honest seam trail for this proof should be:

1. `ingest_validation`
2. `transformation_check`
3. `output_validation`
4. `release_readiness`

This should remain a first-pass proof only.

The seam trail is meant to answer whether one bounded campaign / editorial packet can:

- enter the Airtable surface cleanly
- preserve the campaign-family burden across transformation
- produce governed output that is coherent enough to validate
- reach readiness with the support of explicit review-board context

This proof should not widen into multi-revision stress, contradiction stress, or governed output automation.

## Can Current Packet Logic Carry This Burden Without Pre-Adjustment?

Planning-level answer: yes, this bounded proof should be attempted on the current packet logic without pre-adjustment.

That answer is narrow and provisional.

It does not mean the current packet logic is already proven sufficient for campaign / editorial work in general.

It means only that current repo-local truth still supports one honest attempt inside the existing packet logic because:

- the repo still appears to share one packet-and-seam shape across distinct governing pressures
- campaign / editorial burden has already been described in repo-local notes without yet earning a new schema
- the proposed proof is deliberately bounded to one mini-family, one review board, and one first seam trail

The live proof question is therefore not whether campaign / editorial deserves pre-adjustment in theory.

The live proof question is whether one bounded campaign / editorial packet can be declared and carried honestly enough on the current surface to justify later findings.

## Recommended Next Artifact

The next artifact after this planning doc should be one bounded campaign / editorial Airtable execution plan or proof run prepared directly from this shape.

No narrower note is likely to carry the burden honestly, because the repo already has the comparison and pressure notes needed to explain why this mode matters. What was missing was one concrete proof shape tight enough to execute later without widening scope.

## Explicitly Out Of Scope

This proof plan does not authorize:

- schema revision
- Airtable configuration revision
- governed output automation
- multi-packet campaign systems
- generalized approval-system design
- prompt-artifact continuity reopen unless later execution evidence clearly forces that conclusion
- generalized second-mode rollout
- a broader campaign/editorial implementation move

The purpose here is only to define one bounded next proof.
