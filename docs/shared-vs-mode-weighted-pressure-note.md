# Shared vs Mode-Weighted Pressure Note

## Purpose

This note provides a planning-level comparison of the four worked-example workflow modes now present on `main` through one narrow lens:

- shared pressures
- mode-weighted pressures
- still-unearned claims

Its purpose is to say only what the current examples now appear to support without turning that support into schema, validator expansion, runtime language, or final workflow taxonomy claims.

## What Current Examples Now Support Across All Four Modes

The repo now has four worked-example workflow modes inside the same packet-and-seam structure:

- `SKU-driven product imagery`
- `Collection / Merchandising`
- `marketing / message-driven`
- `brand campaign / editorial`

That is enough to support a stronger planning-level comparison than the repo could make when fewer modes were in view.

The strongest current claim is not that the four modes have now resolved the architecture. The strongest current claim is that the examples now support a clearer distinction between pressures that appear to hold across all four modes and pressures that appear to shift in weight from one mode to another.

The examples still appear to live inside one shared packet-and-seam structure. Across all four, ingest-facing context and boundaries remain legible, constraints are expected to carry across later work, governed outputs remain a distinct question, and release-readiness remains narrower than approval itself.

## Pressures That Appear Shared

Several pressures now appear to be shared across all four modes at a planning level.

First, each mode still depends on declared context and declared boundaries rather than asking later work to infer the governing frame from outputs alone.

Second, each mode still treats governed outputs as something that must remain usable for controlled downstream purposes rather than merely plausible as isolated artifacts.

Third, each mode still appears to require that essential subject-defining, identifier-bearing, message-bearing, or review-relevant signals do not silently disappear as work moves across the chain.

Fourth, bounded creative discretion still appears to be shared across all four modes. The allowed range changes, but the repo's current examples do not support unconstrained creative latitude in any of the modes.

Fifth, release-readiness still appears distinct from approval itself across all four modes. The current examples support readiness to present for approval, not automated approval.

These claims remain provisional, but current repo-local evidence supports them as the strongest shared tendencies now visible.

## Pressures That Appear Mode-Weighted

The strongest current differences appear to be differences of governing weight rather than proof that the repo already needs separate top-level architectures for each mode.

In `SKU-driven product imagery`, governing pressure still appears to weight single-item truth, comparability, and narrow variation most heavily. Product fidelity carries more of the burden here than in the other modes.

In `Collection / Merchandising`, governing pressure still appears to weight grouped-set truth, adjacency, spacing, pairing, and relational legibility most heavily. The strongest burden is whether the set reads as one governed merchandising arrangement rather than as one hero SKU or a loosened room story.

In `marketing / message-driven`, governing pressure still appears to weight targeted communication purpose, message persistence, and bounded promotional variation most heavily. The strongest burden is whether the asset family remains one controlled communication set whose promotional purpose survives across the intended surfaces.

In `brand campaign / editorial`, governing pressure still appears to weight campaign-family continuity, bounded expressive range, and reviewable coherence most heavily. The strongest burden is whether multiple assets remain recognizably part of one campaign family without collapsing back into stricter single-item or message-set logic.

What changes across the four modes is not the existence of governing pressure itself, but which pressure carries the most weight and how much interpretive range can remain acceptable while outputs still stay governable.

## What Remains Unearned

Several stronger claims still appear unearned on current `main`.

The repo has not yet earned schema hardening from this comparison. The examples now support a pressure comparison, not a field system.

The repo has not yet earned stronger claims about fixed inheritance between shared and mode-specific constraints.

The repo has not yet earned a settled claim about exactly where review artifacts should be treated most sharply across the four modes.

The repo has not yet earned a final statement that the current four modes are a closed or final workflow taxonomy.

The repo has not yet earned stronger claims that every currently visible difference should be interpreted as durable architectural structure rather than as provisional example pressure.

## Why This Does Not Yet Settle Universal Layer vs Workflow-Stream Layer vs Brand-Overlay Layer

The current examples make the three-layer read more useful, but they do not yet settle it.

The repo now has better support for saying that some pressures appear broadly shared, some appear workflow-stream-weighted, and some appear more likely to belong to brand or client overlays. Even so, the current evidence still does not justify a stronger claim that those layer boundaries are now crisp, settled, or universally assignable.

That matters because the visible distinctions are still being read from provisional worked examples rather than from a hardened ontology. The four-mode comparison sharpens the planning question, but it does not yet close it.

For now, the strongest honest claim is that the three-layer read remains increasingly useful and increasingly legible, while still provisional.

## Out Of Scope

This note does not do the following:

- schema design
- field catalogs
- validator expansion
- runtime or orchestration design
- approval-policy automation
- new seam design
- final workflow taxonomy settlement
- claims that the current examples close the architecture
