# Attachment-Input Decision Note // SKU-Driven Furniture v1

## Current Decision

Current decision:

- hold
- Option A is the chosen path
- no immediate structural adjustment, robustness rerun, cross-mode check, or generalized attachment-input doctrine is authorized

The repo therefore remains at hold / planning-only posture on the attachment-input question.

## Why The Decision Is Hold

`PKT-SKU-006` earned real `spec-earned pressure`.

What that means, narrowly, is:

- attachment-backed product image input appeared to materially improve control in one bounded SKU-driven comparison
- the result suggests visual anchoring may be load-bearing for product-truth fidelity in this kind of SKU-driven case

What it does **not** mean is:

- the repo has earned a structural adjustment now
- `workflow_packets.product_image_lookup` should already be adopted as a first-class SKU-mode convention
- a robustness rerun should already be run
- a cross-mode check should already be run
- generalized attachment-input doctrine is now justified

The reason for hold is simple:

- the evidence still comes from one comparison packet
- the evidence still comes from one generated result

That is enough to preserve the pressure as repo truth.

It is not enough to justify implementation, structural change, or broader doctrine.

## What Hold Means

Hold means:

- preserve the finding
- make no structural change now
- do not adopt `product_image_lookup` as a first-class SKU-mode convention yet
- do not run a robustness rerun yet
- do not run a cross-mode check yet

This does **not** treat the pressure as unimportant.

It preserves the pressure as earned repo-local truth while declining implementation for now.

## Trigger Conditions For Reopening The Question

This question should be reopened only if at least one of the following occurs:

- a second independent SKU-mode comparison reproduces the same directional improvement
- prompt-only generation demonstrably fails product-truth sufficiency while attachment-backed generation restores governability
- an attachment-backed input path becomes necessary to preserve candidate-output attribution or product-truth fidelity in a later bounded run
- a non-SKU mode shows the same pressure clearly enough to challenge SKU-only framing

Absent one of those triggers, the current decision remains in force.

## Out Of Scope

This note does not authorize:

- generalized attachment-input doctrine
- schema redesign
- multi-packet rollout
- cross-mode rollout
- runtime/orchestration architecture
- reopening prompt-artifact continuity unless its own triggers fire independently
- using Run 1 collateral-generation behavior as justification for attachment-input decisions

It also does not reopen the full options comparison. That work is already recorded elsewhere.
