# SKU Workflow Diagnosis Example

This example applies the workflow diagnosis memo to the abstract SKU workflow discovery case.

## Workflow Snapshot

- A modular, repeatable workflow centered on coverage, consistency, and scalable batch output
- Observed mode is more repeatable than high-variance, though still dependent on limited item-specific adjustment
- The current operational frame combines structured intake, repeatable presentation rules, validation-heavy review, and explicit exception handling

## Core Findings

### Intent

- Business intent is dominant around coverage, consistency, and throughput, while creative intent is present mainly as presentation quality control
- The workflow benefits from a clear split between expression and standardization, but that balance is still managed more implicitly than formally

### Inputs

- Supplied parameters carry more structural weight than contextual instructions
- Source materials are treated as operational inputs to a repeatable system rather than as inspiration for interpretive direction

### Constraints

- Hard requirements are strong and legible around coverage, conformity, and accurate representation
- Bounded discretion exists, but it is intentionally narrow and justified mainly by clarity rather than creative expansion

### Orchestration

- The workflow is batch-oriented and repeatable, with exception routing rather than open-ended exploration driving most variation
- Handoffs are clearer than in a campaign case, but exceptions still reveal where the repeatable model strains

### Outputs

- Outputs are expected to be comparable and coverage-complete more than highly expressive
- Exception cases already behave like a secondary output class even though outputs remain only lightly differentiated in the ontology

### Governance

- Governance is primarily validation-heavy, with human approval focused on sufficiency and exception legitimacy
- Override is structurally important because repeatable systems still encounter real edge cases that cannot be absorbed silently

## Key Tensions

- The workflow aims for repeatability, but some item-specific adjustments still need judgment that is not fully formalized
- Inputs and constraints are mostly distinct, yet some structured parameters begin to function like hard requirements in practice
- Governance is clearer than in a campaign workflow, but validation and orchestration still overlap at exception checkpoints

## Working Diagnosis

- The workflow is structurally strong where repeatable presentation, coverage rules, and validation criteria are explicit
- It becomes less stable where recurring exceptions expose unmodeled variation or where narrow discretion is handled informally rather than structurally

## Advisory Priorities

- Clarify which supplied parameters are true inputs versus effectively mandatory constraints
- Strengthen the boundary between normal batch validation and formal exception handling
- Make the role of bounded discretion explicit so item-specific adjustments remain controlled rather than ad hoc

## Open Questions

- How much item-specific adaptation should remain inside the standard workflow before a separate exception path is triggered?
- Which exception patterns are frequent enough to justify redesign of the repeatable model itself?
- How much creative discretion is actually needed to preserve quality without weakening comparability?
