# Airtable Cycle 4 Contradiction Test Plan // SKU-Driven Furniture v1

## Purpose Of Cycle 4

Cycle 4 is a bounded next-step plan for the current Airtable proof.

Its purpose is to test contradiction density inside the existing packet-level composition surface without widening the current v1 proof into schema revision, configuration revision, or a larger composition subsystem.

The cycle is meant to distinguish:

- additive-density fragility versus contradiction-density fragility
- survivability versus genuine stability
- craft-recoverable fragility versus a structurally earned need for a more explicit composition surface

## Why Cycle 4 Is Warranted Now

Cycle 1 established that the v1 graph and deterministic seam chain could carry one bounded packet with one bounded revision loop.

Cycle 2 established that prompt composition had become a live-tested pressure and that the smallest honest local remedy was packet-level prompt artifact plus packet-level composition notes.

Cycle 3 established that the current packet-level composition surface still survives a denser packet burden, but that flattening pressure rises quickly with density even when graph adequacy still holds.

That means the next smallest honest question is no longer only whether additive density alone increases fragility. The next bounded question is whether contradiction density introduces a sharper failure mode than additive density did, and whether contradiction pressure introduces a priority-resolution burden that additive density alone does not expose.

This plan is therefore warranted as a bounded next-cycle test plan, not as a spec rewrite and not as a new explanatory note triggered by completeness pressure.

## What Cycle 4 Will Test That Cycles 1-3 Did Not

Cycles 1-3 established:

- seam survivability
- prompt-composition wedge visibility
- rising fragility under additive density

Cycle 4 should test one narrower question that those cycles did not yet isolate:

- whether the current packet-level composition surface remains usable when the packet must carry not just more burden, but burden that pulls in conflicting directions unless composition resolves it deliberately

The test question is not whether contradiction density is already proven to be the dominant failure source.

The test question is whether contradiction density produces a meaning-loss pattern materially sharper than the additive-density pressure already observed in cycle 3.

## Bounded Test Shape

Cycle 4 should remain inside the existing SKU-driven Furniture v1 Airtable proof.

The bounded shape should be:

1. use the existing seven-table v1 graph
2. use the existing packet-level composition surface already present from cycle 2
3. introduce one bounded stress packet whose burden includes controlled contradiction pressure inside the packet's carried meaning
4. run one first-pass transformation and one first-pass `output_validation`
5. if the first pass fails, run one local composition revision and one second-pass transformation plus second-pass `output_validation`
6. stop before `release_readiness` and before any gate step unless contradiction testing unexpectedly cannot be interpreted without them

No new table is pre-authorized.

No proof-spec update is pre-authorized.

No Airtable configuration update is pre-authorized.

No generalized composition subsystem is being introduced.

## What Counts As Contradiction Density In This Repo Context

For this cycle, contradiction density means a packet that remains within current v1 scope and contains valid packet elements that compete for priority, emphasis, or interpretation in a way that forces deliberate composition resolution.

In this repo context, that can include tension such as:

- product truth pulling toward one visual reading while approved directional references pull toward another
- scoped rules that remain individually valid but become difficult to satisfy together without careful composition
- bounded creative discretion that is still legitimate but can be over-read in a way that weakens required output discipline
- required output expectations that are still in scope but become easier to underspecify when other packet signals compete for emphasis

The contradiction used in cycle 4 should be deliberate, bounded, minimally sufficient, and interpretable; it should not be arbitrary invalidity and should not be accidental ambiguity masquerading as contradiction.

The point is to test controlled contradiction pressure inside otherwise valid packet meaning, not to manufacture failure by breaking v1 assumptions.

## Explicitly Out Of Scope

Cycle 4 does not pre-authorize:

- schema revision
- Airtable field or table reconfiguration
- a new composition table
- a generalized prompt-composition framework
- runtime orchestration design
- approval-policy automation
- generalized conclusions about other workflow modes
- treating external control-surface behavior as repo-local system design

Connector, safety, or request-surface issues should still be recorded separately from proof-surface findings if they materialize during execution.

## Evaluation Thresholds

### Findings-Only Outcome

A findings-only outcome is earned if cycle 4 shows that contradiction density creates a sharper first-pass failure or stronger ambiguity, but the packet can still be recovered through one bounded local composition revision inside the current packet-level surface.

That outcome would mean:

- the graph still carried the burden
- the current packet-level remedy still survived
- the main result belongs in a new findings artifact rather than in spec or configuration changes

### Spec / Configuration-Earned Outcome

A spec- or configuration-earned outcome is earned only if cycle 4 shows that contradiction density cannot be kept legible inside the current packet-level composition surface without structure that the current proof does not explicitly carry.

Signals may include:

- repeated ambiguity that cannot be made legible through one bounded local revision
- failure modes that show the current packet-level surface is too implicit rather than merely too delicate
- evidence that a more explicit composition surface is needed before another honest cycle can be run

That outcome would justify a follow-on planning or spec/configuration decision. It would not authorize the change by itself inside the cycle run.

### Insufficient-Evidence Outcome

An insufficient-evidence outcome is earned if cycle 4 cannot cleanly distinguish contradiction-density pressure from other causes.

That may include:

- a packet that is merely denser rather than meaningfully more contradictory
- platform or safety limitations that dominate the run
- a first-pass failure that cannot be attributed clearly enough to contradiction pressure versus local builder craft

If this happens, the cycle should end with a findings artifact that says the evidence did not cleanly decide the question.

## Expected Artifacts After Execution

If cycle 4 is run, the expected repo-local artifact is one new findings document recording the empirical result of the contradiction-density cycle.

That findings document should stay within the existing Airtable findings posture:

- observable base state
- what was tested
- what worked
- what failed
- where meaning became ambiguous
- whether the result stayed findings-only or instead earned follow-on spec/configuration pressure

No additional planning note is automatically earned unless the cycle result shows that current notes can no longer carry the burden.
